# 2492. Minimum Score of a Path Between Two Cities

**Сложность:** Medium

**Ссылка:** https://leetcode.com/problems/minimum-score-of-a-path-between-two-cities/

## Условие

Дано `n` городов (пронумерованы от 1 до `n`) и массив дорог `roads`, где
`roads[i] = [ai, bi, distancei]` — двусторонняя дорога между городами `ai` и
`bi` длиной `distancei`.

**score** маршрута = вес самого короткого ребра на этом маршруте.

Нужно вернуть минимально возможный score среди всех маршрутов из города `1`
в город `n`. По маршруту можно проходить по дорогам и городам **сколько угодно
раз** (маршрут не обязан быть простым путём).

## Примеры

**Пример 1:**
- Ввод: `n = 4`, `roads = [[1,2,9],[2,3,6],[2,4,5],[1,4,7]]`
- Вывод: `5`
- Пояснение: маршрут `1 -> 2 -> 4` даёт score `min(9, 5) = 5`. Меньше получить нельзя.

**Пример 2:**
- Ввод: `n = 4`, `roads = [[1,2,2],[1,3,4],[3,4,7]]`
- Вывод: `2`
- Пояснение: маршрут `1 -> 3 -> 4 -> 1 -> 2` даёт score `min(4, 7, 4, 2) = 2`.

## Ограничения

- `2 <= n <= 10^5`
- `1 <= roads.length <= 10^5`
- `roads[i].length == 3`
- `1 <= ai, bi <= n`
- `ai != bi`
- `1 <= distancei <= 10^4`
- Нет повторяющихся рёбер
- Города `1` и `n` гарантированно связаны

## Решение

Ключевой трюк: по дорогам можно ходить сколько угодно раз и возвращаться,
поэтому конкретный маршрут искать **не нужно**. Любое ребро связной компоненты,
где лежат города `1` и `n`, можно «зацепить» в каком-нибудь маршруте `1 -> n`
(дойти до него, переехать, вернуться, доехать до `n`).

Значит ответ = **минимальный вес ребра в компоненте, достижимой от города 1.**

1. Строим неориентированный граф как список смежности (`Map<city, соседи[]>`),
   каждое ребро добавляем в обе стороны.
2. Обходим компоненту от города `1` (DFS на явном стеке).
3. Копим минимум по **каждому** встреченному ребру — даже если сосед уже
   посещён (`visited` решает только «идти ли дальше», но не влияет на min).
4. Возвращаем накопленный минимум.

## Сложность

- Время: O(V + E) — обход графа
- Память: O(V + E) — список смежности + стек/visited

## Заметки

Наивный `Math.min(...all distances)` неверен: в графе могут быть **отдельные
компоненты** с более дешёвыми, но недостижимыми от города `1` рёбрами. На
большом тесте (`bif-mock`) такой минимум даёт `287` (ребро `13–23` в чужой
компоненте), тогда как правильный ответ `418` — минимум по достижимой компоненте.

Рекурсивный DFS тут не используем: при `n` до `10^5` глубина рекурсии может
переполнить call stack, поэтому ведём стек вручную.
