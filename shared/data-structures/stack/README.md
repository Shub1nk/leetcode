# Stack

Стек (LIFO — last in, first out) как **фасад над `Deque`**. Своей структуры данных не держит: композиция, внутри приватный `Deque`, наружу торчит только стековый API.

## API

```ts
const st = new Stack<number>();

st.push(x);     // положить на вершину
st.pop();       // снять с вершины → T
st.peek();      // посмотреть вершину → T
st.size;        // количество элементов
st.isEmpty();   // пустой ли
```

На пустом стеке `pop`/`peek` **бросают** `StackUnderflowError` (сообщение экспортируется как `STACK_ERROR_EMPTY`). Это осознанно строже базового `Deque`, который на пустом отдаёт `undefined`: у стека `pop`/`peek` всегда возвращают `T`, и вызывающему не нужно проверять `undefined`.

## Реализация

Композиция, а не наследование. `extends Deque` протащил бы наружу `pushFront`/`popBack` и т.п. — и LIFO-инвариант перестал бы защищаться. Поэтому `Deque` спрятан приватным полем, а методы делегируют в **один конец** (хвост):

- `push` → `deque.pushBack`
- `pop` → `assertNotEmpty()` + `deque.popBack()!`
- `peek` → `assertNotEmpty()` + `deque.peekBack()!`
- `size` / `isEmpty` → проброс

`assertNotEmpty` даёт throw-контракт, `!` после него снимает `undefined` из типа (стек гарантированно непуст).

## Сложность

- `push` / `pop` / `peek`: O(1) амортизированно (наследует от `Deque`).
- Память: O(n).

## Заметки

- Композиция «has-a», не наследование «is-a»: стек *использует* дек, а не *является* им.
- Работаем строго за хвост (`*Back`) — конец выбран произвольно, важна лишь консистентность push/pop/peek на одном конце.
